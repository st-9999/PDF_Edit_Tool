## 1. プロダクト概要

ブラウザ上で動作する、**PDFの閲覧＋編集**ができる Web アプリケーション。
サーバーにファイルを送らず、**すべての処理をブラウザ内（クライアントサイド）で完結**させる。

- **主軸**: 編集（並び替え・回転・削除・抽出・分割・結合・テキスト編集）
- **将来**: しおり（ブックマーク）作成、自動生成
- **対応**: PC専用（デスクトップ、マウス操作・ドラッグ&ドロップ前提）

### プライバシー上の利点
ファイルがネットワークに出ないため、機密文書でも安心して扱える（これを売りにできる）。

## 2. 技術構成

| 項目 | 採用 | 備考 |
|---|---|---|
| フレームワーク | Next.js（`output: 'export'` 静的エクスポート） | GitHub Pages は静的ホスティング専用のため |
| 言語 | TypeScript | |
| UI | shadcn/ui + Tailwind CSS | |
| PDF表示 | `pdf.js`（`react-pdf` 等のラッパー） | ページ単位の遅延レンダリングが可能 |
| PDF編集 | `pdf-lib` | 並び替え・回転・削除・抽出・分割・結合 |
| 状態管理 | Zustand（候補） | Undo/Redo 履歴を扱うため |
| 重い処理 | Web Worker | UIスレッドのブロック回避 |
| デプロイ | GitHub Pages（GitHub Actions でビルド&公開） | |

### GitHub Pages 制約からくる設計方針
- サーバー処理は一切なし。OCR・自動しおり生成も**ブラウザ内**で行う必要がある。
- ルーティングは静的。動的ルートを使う場合は `generateStaticParams` 等で事前生成。
- `basePath` をリポジトリ名に合わせる（例: `/pdf-edit-tool`）。

## 3. 機能要件

### 3.1 ビューア（共通基盤）
- PDFファイルのドラッグ&ドロップ / ファイル選択で読み込み
- ページ送り、連続スクロール表示、ズーム（フィット／幅／%指定）
- サムネイル一覧（編集操作の起点）
- ページ番号ジャンプ
- **テキスト検索**（v1。pdf.js のテキストレイヤを使用）
  - 検索語のハイライト・ヒット間移動
  - テキストの**部分選択・全選択・コピー**（テキストレイヤ上で選択操作を可能にする）

### 3.2 編集機能（v1 スコープ）
| 機能 | 内容 | 実装の見通し |
|---|---|---|
| ページ並び替え | サムネイルの D&D で順序変更 | 容易（pdf-lib でページ再構成） |
| ページ回転 | 90度単位の回転 | 容易 |
| ページ削除 | 不要ページの削除 | 容易 |
| 抽出・分割 | 選択ページを新規PDFとして保存／指定位置で分割 | 容易 |
| 複数PDF結合 | 複数ファイルを読み込み1つに結合 | 容易（※大容量時はメモリ注意、後述） |
| Undo/Redo | 編集操作の取り消し・やり直し | v1から。状態管理の設計が必要 |

> **※ PDF内テキスト編集機能は廃止（スコープ外）**。
> 既存テキストの“その場編集”はライブラリの正式サポート外で難易度が極めて高いため、本プロダクトでは扱わない。詳細は「6.1」を参照。

### 3.3 しおり（ブックマーク）
左ペインの「しおり」タブで扱う（5.2 参照）。段階導入。
- **v1（閲覧）**: 既存PDFのアウトライン（目次）をツリー表示し、クリックで該当ページへジャンプ
- **v2（編集）**: しおりの追加・リネーム・階層変更・削除 → 保存時にPDFのOutlineへ書き戻し
- **v3（自動生成）**: フォントサイズ等のヒューリスティック／目次構造流用／テキストパターン等
  - 方式は別途要件定義する `[要決定]`

### 3.4 出力・保存
- **その都度、ファイル名と保存場所を指定して保存**（保存ダイアログ）
- **上書き保存**（編集元のファイルへ直接書き戻す）
- 分割時は複数ファイルを順次保存
- **ブラウザ依存（重要）**: 保存場所指定・上書き保存は **File System Access API** を使用し、**Chromium系（Chrome/Edge）のみ対応**。**Firefox は非対応**のため、ブラウザの能力に応じて挙動を切り替える（下表）。

| 保存機能 | Chrome / Edge | Firefox |
|---|---|---|
| 保存場所をその都度指定 | ✅ 保存ダイアログ | △ ダウンロード先に保存（場所指定不可） |
| 上書き保存（元ファイルへ） | ✅ | ❌ 不可（再ダウンロードで代替） |

> Firefox では File System Access API が無いため、`<a download>` 経由のダウンロード保存にフォールバックする。UI上は対応状況を明示する。

## 4. 非機能要件

- **対象ブラウザ**: 最新の **Chrome / Edge / Firefox** に対応。
  - pdf.js・pdf-lib はいずれも3ブラウザで動作する。
  - **保存系（保存場所指定・上書き保存）は Chromium系のみフル対応**、Firefox はダウンロード保存へフォールバック（3.4 参照）。
  - `performance.memory` 等の計測API は Chromium のみ → 計測は Chromium 基準で実施。
- **想定規模**: 大（数千ページ / 100MB超）→ **「6. 重要な技術的論点」のメモリ制約を必読**
- **オフライン**: 不要（PWA化はしない）
- **国際化**: 日本語UIのみ
- **アクセシビリティ / キーボード操作**: `[要決定]`（基本的なキーボード操作は考慮）

## 5. 画面構成（UI設計）

PC専用・編集メイン。**3ペイン（トップバー＋左ペイン＋メインビューア）＋ステータスバー**構成。UIは shadcn/ui + Tailwind。

```
┌─────────────────────────────────────────────────────────────┐
│ [トップバー] ロゴ │ 開く 結合 │ ↶Undo ↷Redo │ 🔍検索 │ 保存▼ │
├──────────────┬──────────────────────────────────────────────┤
│ [サムネ|しおり]│ [中央：メインビューア]                       │ ← 左ペインはタブ切替
│ ── サムネ ── │   ┌────────────────────────┐                 │
│  □1 ⠿        │   │     ページ拡大表示       │                 │
│  □2 ⠿ [選択] │   │  (テキストレイヤ付き)    │ ← 検索/部分・全選択
│  □3 ⠿        │   └────────────────────────┘                 │
│  …D&Dで並替  │   ◀ 3 / 248 ▶  ズーム[─●──]100% ⊟全体        │
├──────────────┴──────────────────────────────────────────────┤
│ [ステータスバー] 248ページ │ 12.4MB │ 未保存● │ 処理中…[×]    │
└─────────────────────────────────────────────────────────────┘
```

### 5.1 トップバー（ツールバー）
- ファイル: 開く / 複数PDF結合（ファイル追加）
- 編集: Undo（↶）/ Redo（↷）— 常時表示、無効時グレーアウト
- ページ操作: 回転（左/右）/ 削除 / 抽出（選択を新規PDFへ）/ 分割（指定位置）。**ページ選択時にアクティブ化**（文脈依存）
- 検索: ビューア上に検索バーを開く
- 保存（`保存▼` ドロップダウン）: 「名前を付けて保存」「上書き保存（Chrome/Edgeのみ）」「分割して保存」

### 5.2 左ペイン：**タブ切替（サムネイル ⇄ しおり）**
左ペインは2つのビューをタブ（`Tabs`）で切り替える。幅は `Resizable` で可変。

**［サムネイル］タブ**（編集の起点）
- 各ページのサムネ＋ページ番号＋ドラッグハンドル（⠿）
- **D&Dで並び替え**、**複数選択**（クリック / Shift範囲 / Ctrl個別）→ 選択ページへ一括操作
- 右クリックで `ContextMenu`（回転 / 削除 / ここで分割 / 抽出）
- 回転状態・削除予定（半透明）をサムネに反映

**［しおり］タブ**（アウトライン）
- PDFのアウトライン（目次）を**ツリー表示**、クリックで該当ページへジャンプ
- **v1**: 既存しおりの**閲覧・ジャンプ**（ビューア機能）
- **v2**: しおりの**追加・リネーム・階層変更・削除**（編集UI）→ 保存時にPDFのOutlineへ書き戻し
- **v3**: しおり**自動生成**（方式は別途要件定義）
- しおりが無いPDFでは空状態（v2以降で「ここから作成」を案内）

### 5.3 中央ペイン：メインビューア
- 選択ページを拡大表示。**テキストレイヤを重ね、部分選択・全選択・コピー可**
- ズーム（`Slider` ＋ ％入力 ＋ 幅合わせ/全体表示）、連続スクロール / 単ページ送り切替
- 検索バー: ヒット件数（3/12）、前後移動、ハイライト

### 5.4 ステータスバー
- 総ページ数 / ファイルサイズ / **未保存インジケータ（●）**
- 重い処理中は**進捗バー＋キャンセル**（§9.5 と連動）

### 5.5 画面状態
| 状態 | 表示 |
|---|---|
| 初期（未読込） | 中央に大ドロップゾーン「PDFをドラッグ / クリックで選択」 |
| 読み込み中 | スケルトン＋プログレス |
| 編集中 | 上記3ペイン |
| 処理中（保存/結合等） | オーバーレイ＋進捗＋キャンセル |
| エラー | トースト（`Sonner`） |
| 保存 | `Dialog`: 名前＋保存先（Chrome/Edge）/ 名前のみ（Firefox） |

### 5.6 shadcn/ui コンポーネント対応
| UI要素 | コンポーネント |
|---|---|
| 左ペイン切替 | `Tabs` |
| ツールバー操作 | `Button`, `Tooltip`, `DropdownMenu` |
| 右クリックメニュー | `ContextMenu` |
| 保存/破棄確認 | `Dialog`, `AlertDialog` |
| 検索バー | `Input` + `Button` |
| ズーム | `Slider` + `Input` |
| 通知 | `Sonner` |
| しおりツリー | （shadcn外）独自Treeコンポーネント or `Accordion` ベース |
| サムネD&D | （shadcn外）`dnd-kit` |
| ペイン分割 | `Resizable` |

### 5.7 操作・ショートカット
- Ctrl+Z/Y（Undo/Redo）、Ctrl+S（保存）、Ctrl+F（検索）、Delete（選択削除）
- 未保存のままの離脱は `beforeunload` で警告
- テーマ: ライト/ダーク対応

## 6. 重要な技術的論点（要・意思決定）

エンジニアとして、計画段階で正直にお伝えすべき2点です。仕様の現実性に直結します。

### 6.1 「PDF内テキスト編集」→ 廃止（スコープ外）✅ 決定済み

PDFは、ワープロのような「編集可能な段落」ではなく、**座標上に配置されたグリフ（文字図形）の集合**としてテキストを保持します。`pdf-lib` を含む主要ライブラリは、**既存テキストの“その場編集”を正式サポートしていません**。真の本文編集（Acrobat相当）はコンテンツストリーム解析・フォント抽出/サブセット化が必要で、完全再現は極めて困難です。

**決定（2026-05-29）**: 難易度とリスクが見合わないため、**PDF内テキスト編集機能は廃止**し、本プロダクトのスコープから除外する。
（将来、注釈・テキスト追記のような“オーバーレイ方式”の要望が出た場合は、別機能として改めて要件定義する。）

### 6.2 「大（数千ページ / 100MB超）」× 完全クライアント処理のメモリ制約 ✅ 方針決定済み

**決定（2026-05-29）**: v1は「中規模（〜数百ページ／数十MB）で快適動作」を品質保証ラインとし、大規模はベストエフォート＋上限を計測して明文化する**段階的アプローチ**を採用する。具体的な大規模対応の設計は下記「9. 大規模対応のアプローチ（将来）」を参照。

- **閲覧**は問題になりにくい: pdf.js はページ単位の遅延レンダリングが可能。
- **編集**が課題: `pdf-lib` は**ドキュメント全体をメモリに展開**し、保存時に再シリアライズします。100MB超や数千ページ、さらに複数ファイル結合では、**ブラウザのタブのメモリ上限（おおむね数GB）に達するおそれ**があります。
- v1 の緩和策（設計に織り込む）:
  - Web Worker での処理 + 進捗表示 / キャンセル
  - 結合・抽出はストリーミング志向の実装、不要オブジェクトの早期解放
  - File System Access API で大きな出力を直接ファイルへ書き出し
  - 入力サイズに応じた**事前警告**と推奨上限の提示
- それでも**完全クライアントには物理的な上限**があります（明文化して提示する）。

## 9. 大規模対応のアプローチ（将来 / v2以降）

完全クライアント処理で「数千ページ／100MB超」を快適に扱うための設計指針。**v1では実装せず、必要になった段階で導入**する。考え方は「全体を一度にメモリへ載せない」こと。

### 9.1 計測（まず最初にやること）
- 代表的な操作（読込・結合・抽出・保存）について、ファイルサイズ／ページ数を変えて**メモリ使用量と所要時間を計測**し、安全に動く**上限値を実測で確定**してUIに明示する。
- `performance.memory`（Chrome）や処理時間ログで回帰を監視。CIに簡易ベンチを入れる。

### 9.2 表示（pdf.js 側）
- **ページ仮想化**: 画面に見えているページ＋前後数ページだけレンダリングし、離れたページの canvas は破棄。
- レンダリング解像度をズーム率に追従させ、過剰な高解像度描画を避ける。
- サムネイルは低解像度でキャッシュ上限を設ける（LRU で破棄）。

### 9.3 編集（pdf-lib 側 — ここが本丸）
`pdf-lib` は全体展開型のため、大規模では次のいずれかへ移行する。

| アプローチ | 内容 | 長所 / 短所 |
|---|---|---|
| **A. Web Worker 隔離** | 編集処理をワーカーへ。`Transferable`（ArrayBuffer）で受け渡し、UIスレッドを止めない | 比較的低コスト。メモリ総量は減らない |
| **B. 差分/遅延保存** | 元バイト列を保持し、変更（並び替え・回転・削除）を**操作ログ**として持つ。保存時にのみ適用 | メモリ効率◎。Undo/Redoとも相性が良い |
| **C. インクリメンタル更新** | PDFの追記更新（incremental update）を使い、元ファイルへ差分だけ追記 | 巨大ファイルの保存が軽い。実装は高度 |
| **D. ページ単位ストリーム処理** | 結合・抽出をページ単位で逐次コピーし、こまめに解放 | 大容量結合に有効。実装やや複雑 |
| **E. WASM系の採用** | より省メモリな WASM ベースのPDFエンジンへ部分移行 | 性能◎。バンドルサイズ・ライセンス・実装コスト大 |

**推奨ロードマップ**: まず **A（Worker隔離）** で体感を改善 → 次に **B（操作ログ＋遅延適用）** をコア設計に据える（Undo/Redoの履歴と統合できる）→ 結合系で詰まれば **D** → 限界が見えたら **C / E** を検討。

### 9.4 出力
- **File System Access API** で出力先を先に確保し、生成しながら書き出す（巨大Blobをメモリに溜めない）。
- 分割保存は複数ファイルを順次書き出し、各ファイル完了ごとにメモリ解放。

### 9.5 UX上のガード
- 推奨上限を超える入力には**警告＋処理見積もり（時間/メモリ）**を提示。
- 重い処理は**進捗バー＋キャンセル**を必須にする。

> 補足: 完全クライアントには物理上限があるため、超大規模を“確実に”保証したい場合のみ、将来オプションとしてサーバー/WASMワーカーでの処理を別途検討する（GitHub Pages の制約を外す判断が必要）。
